Understanding Django - The Theory

What is Django?

Django is like a construction kit for websites. Imagine you're building a house - Django provides all the pre-made parts (walls, doors, windows) so you can assemble them quickly rather than making everything from scratch. It's written in Python, which is known for being easy to read and write.

Django handles the boring, repetitive tasks for you (like user logins, database connections) so you can focus on the unique parts of your website.

Why Use Django?

  • ‚úÖ Fast Development: Like using prefabricated parts to build a house faster
  • ‚úÖ Secure: Automatically protects against common hacker attacks
  • ‚úÖ Scalable: Can handle small blogs or huge sites like Instagram
  • ‚úÖ Batteries Included: Comes with tools for user accounts, admin panels, and more
  • ‚úÖ Organized: Forces you to structure your code neatly

Key Django Concepts Explained Simply

P

Project

The entire website container. Like a house that contains all rooms. When you start a new Django project, it's like buying a plot of land and getting the basic structure (foundation, outer walls) set up.

A

Apps

Modular components that do specific jobs. Like rooms in a house - kitchen (cooking), bedroom (sleeping), bathroom (washing). In our project: accounts (logins), products (items for sale), core (shared features).

M

Models

Blueprints for your database. Like forms you fill out - each field is a piece of information. A User model might have: name (text), age (number), join date (date). Django converts these to database tables.

Step 7: Database Setup

Migrations - Database Blueprints

Django uses a system called migrations to manage database changes. Think of it like:

  1. You design how your database should look (models)
  2. Django creates "migration" files - instructions for building that structure
  3. Running migrations actually builds the database tables

This lets you change your database structure safely over time.

# Run migrations to create database tables

What Happens:

  • Django checks all installed apps for migration files
  • It creates the SQLite database file (db.sqlite3) if it doesn't exist
  • It runs all migrations to create the initial database tables

Result: Your database now has tables for users, sessions, etc. that Django needs to work.

Understanding JWT Authentication

What is JWT? (Cookie-Free Authentication)

Traditional websites use cookies to remember logged-in users. JWTs (JSON Web Tokens) are a modern alternative that works especially well for APIs. Here's how it works:

1. Login

User sends username/password

‚Üí Server verifies and creates JWT

2. Token

Server sends back a signed token containing user info

3. Requests

User sends token with each request

‚Üí Server verifies token

Why JWT for our Fashionista project? It works perfectly with mobile apps and frontend frameworks like React, which we might add later.

Step 2: Configure JWT Settings

Token Lifetimes Explained

JWTs have two types of tokens:

  • Access Token - Short-lived (minutes/hours) used for regular requests
  • Refresh Token - Longer-lived (days/weeks) used to get new access tokens

This balances security (frequent re-authentication) with user convenience (not logging in constantly).

# Add to settings.py
from datetime import timedelta

REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': (
        'rest_framework_simplejwt.authentication.JWTAuthentication',
    )
}

SIMPLE_JWT = {
    'ACCESS_TOKEN_LIFETIME': timedelta(minutes=60),
    'REFRESH_TOKEN_LIFETIME': timedelta(days=1),
}

Configuration Details:

  • DEFAULT_AUTHENTICATION_CLASSES tells DRF to use JWT by default
  • ACCESS_TOKEN_LIFETIME sets how long access tokens are valid (60 minutes here)
  • REFRESH_TOKEN_LIFETIME sets how long refresh tokens work (1 day here)

Tip: In production, you might set access tokens to 15-30 minutes and refresh tokens to 7-30 days.

Step 3: Create Custom User Model

Why Custom User Model?

Django comes with a built-in User model, but it's better to create a custom one because:

  • We want to use email as the login instead of username
  • We might need to add custom fields later (phone number, profile picture, etc.)
  • It's harder to switch after you've already created the database

Best Practice: Always start projects with a custom user model, even if it's identical to the default at first.

# In accounts/models.py
from django.contrib.auth.models import AbstractUser
from django.db import models

class User(AbstractUser):
    email = models.EmailField(unique=True)  # Email must be unique
    phone = models.CharField(max_length=15, blank=True)  # Optional phone
    
    # Use email instead of username for authentication
    USERNAME_FIELD = 'email'
    
    # When creating superuser, it will prompt for these fields
    REQUIRED_FIELDS = ['username']
    
    def __str__(self):
        return self.email

Key Components:

  • AbstractUser - The base Django user model we're extending
  • unique=True - Ensures no two users can have the same email
  • USERNAME_FIELD - Tells Django to use email for login
  • REQUIRED_FIELDS - Fields needed when creating superusers
  • __str__ - How the user will be displayed in admin

Update Settings.py

We need to tell Django to use our custom user model instead of the default one.

# In Fashionista/settings.py

Step 4: Create Registration API

Registration Flow

When a new user signs up, our API will:

  1. Receive email, username, and password
  2. Validate the data (check email format, password strength)
  3. Create a new user in the database
  4. Return a success message or errors

We'll use Django REST Framework's serializers to handle data conversion and validation.

# Create accounts/serializers.py
from rest_framework import serializers
from django.contrib.auth import get_user_model

# Get our custom user model
User = get_user_model()

class UserRegistrationSerializer(serializers.ModelSerializer):
    # Password field won't be included in response
    password = serializers.CharField(write_only=True)
    
    class Meta:
        model = User
        fields = ['email', 'username', 'password']
    
    def create(self, validated_data):
        # Create and return a new user instance
        user = User.objects.create_user(
            email=validated_data['email'],
            username=validated_data['username'],
            password=validated_data['password']
        )
        return user

Serializer Breakdown:

  • ModelSerializer - Automatically generates fields based on the model
  • write_only=True - Password will be used for creation but not returned in API responses
  • create_user() - Django's method that properly hashes passwords
  • validated_data - Contains already validated input data

Security Note: Never store or return plain text passwords!

Step 5: Create Login API

Login Flow

When a user logs in, our API will:

  1. Receive email and password
  2. Verify credentials against database
  3. If valid, generate JWT tokens
  4. Return tokens to the user
  5. If invalid, return error message

We'll extend DRF's built-in token view to customize the response.

# In accounts/views.py
from rest_framework import generics, status
from rest_framework.response import Response
from rest_framework_simplejwt.views import TokenObtainPairView
from .serializers import UserRegistrationSerializer

class RegisterView(generics.CreateAPIView):
    serializer_class = UserRegistrationSerializer
    
    def post(self, request, *args, **kwargs):
        serializer = self.get_serializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        user = serializer.save()
        return Response({
            "message": "User created successfully",
            "user": serializer.data
        }, status=status.HTTP_201_CREATED)

class LoginView(TokenObtainPairView):
    def post(self, request, *args, **kwargs):
        # Get the response from the original TokenObtainPairView
        response = super().post(request, *args, **kwargs)
        
        # Customize the response
        return Response({
            "access_token": response.data['access'],
            "refresh_token": response.data['refresh'],
            "message": "Login successful"
        }, status=status.HTTP_200_OK)

View Components:

  • RegisterView - Handles POST requests to create new users
  • LoginView - Extends JWT's token view to customize response format
  • raise_exception=True - Automatically returns error responses for invalid data
  • status.HTTP_201_CREATED - Proper status code for resource creation

Note: The parent TokenObtainPairView already handles all the JWT generation logic - we're just customizing how it's returned.

Step 2: Creating Profile Serializers (serializers.py)

What Are Serializers?

Serializers are like translators that:

  • Convert complex data (models) to Python datatypes (JSON)
  • Validate incoming data meets requirements
  • Handle file uploads (like profile pictures)

We need two serializers because password changes require special handling.

# accounts/serializers.py
from rest_framework import serializers
from .models import User

class ProfileSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = [
            'username', 
            'email', 
            'profile_picture', 
            'bio', 
            'phone_number', 
            'birth_date'
        ]
        extra_kwargs = {
            'email': {'read_only': True}  # Prevent email changes
        }

class ChangePasswordSerializer(serializers.Serializer):
    old_password = serializers.CharField(
        required=True,
        style={'input_type': 'password'}
    )
    new_password = serializers.CharField(
        required=True,
        min_length=8,
        style={'input_type': 'password'}
    )

Key Components Explained:

  • ProfileSerializer:
    • ModelSerializer: Auto-generates fields from model
    • extra_kwargs: Additional field settings (making email read-only)
  • ChangePasswordSerializer:
    • Serializer: Basic serializer (not tied to model)
    • style={'input_type': 'password'}: Makes passwords appear as dots
    • min_length=8: Enforces password strength

Step 3: Building Profile Views (views.py)

Understanding the Views

We'll create two main views:

  1. ProfileView: Handles viewing and updating profile info
  2. ChangePasswordView: Manages secure password changes

Both require authentication and use different serializers.

# accounts/views.py
from rest_framework import generics, permissions
from rest_framework.response import Response
from django.contrib.auth import update_session_auth_hash
from .serializers import ProfileSerializer, ChangePasswordSerializer

class ProfileView(generics.RetrieveUpdateAPIView):
    """
    Handles both retrieving and updating user profiles
    """
    serializer_class = ProfileSerializer
    permission_classes = [permissions.IsAuthenticated]

    def get_object(self):
        # Returns the currently authenticated user
        return self.request.user

class ChangePasswordView(generics.UpdateAPIView):
    """
    Handles password changes with old password verification
    """
    serializer_class = ChangePasswordSerializer
    permission_classes = [permissions.IsAuthenticated]

    def update(self, request, *args, **kwargs):
        user = self.request.user
        serializer = self.get_serializer(data=request.data)
        
        # Validate the incoming data
        serializer.is_valid(raise_exception=True)
        
        # Check if old password is correct
        if not user.check_password(serializer.data.get('old_password')):
            return Response(
                {"error": "Wrong password"}, 
                status=status.HTTP_400_BAD_REQUEST
            )
            
        # Set and save the new password
        user.set_password(serializer.data.get('new_password'))
        user.save()
        
        # Keep the user logged in after password change
        update_session_auth_hash(request, user)
        
        return Response(
            {"message": "Password updated successfully"},
            status=status.HTTP_200_OK
        )

View Logic Explained:

  • RetrieveUpdateAPIView: Combines GET (retrieve) and PUT/PATCH (update) operations
  • permissions.IsAuthenticated: Only logged-in users can access
  • get_object: Returns the current user automatically
  • check_password(): Verifies old password matches
  • set_password(): Properly hashes new password
  • update_session_auth_hash: Prevents forced logout after password change
